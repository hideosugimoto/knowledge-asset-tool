# 共通品質ルール

全コマンドファイル（analyze, manual, slide, user-guide 等）で適用される品質ルール。
サブエージェントに委譲する場合も、このファイルを必ず読ませること。

---

## 1. ファクト根拠ルール

1. **実コードのみを根拠にする**。コードで確認できない機能は記載しない
2. **書く前に読む**: 記載する全ての技術的事実（エンドポイント、テーブル名、カラム名、メソッド名、画面名、ボタン名）は、対応するソースコードの存在を確認してから書く。「ありそう」「一般的には」で補完しない
3. **データフロー検証**: シーケンス図やフロー記述を書く前に、ルーティング → Controller → Service/Repository → DB → レスポンスまでの実際のメソッド呼び出しチェーンをコードで確認する。架空のメソッド名・クラス名は使用禁止
4. **フロントエンドUI復元**: 画面構成・フォーム仕様・テーブルカラムを記述する際は、テンプレート（Vue `<template>`, JSX, Blade等）を実際に読み、コンポーネント階層・表示条件（v-if, 条件レンダリング）・バインド先を確認してから書く
5. **ファクトキャッシュの活用**: `.cache/facts-{名前}.yaml` が存在する場合、各エントリの `source` フィールドをソース引用に使用する。ファクトに含まれない事実を本文に書くことは禁止

## 2. ソース引用ルール

技術的事実（コントローラー名、メソッド名、テーブル名、エンドポイント、ビジネスルール）を記述する際は出典を明記する。

### 引用形式

| コマンド種別 | 引用形式 | 例 |
|------------|---------|---|
| analyze, analyze-full, analyze-rag, manual, test-spec, analyze-slide | `(file:line)` 形式（本文に表示） | 認証は JWT で行われる (app/Http/Middleware/Auth.php:23) |
| user-guide, quick-ref, analyze-share | `<!-- source: file:line -->` 形式（HTMLコメント、不可視） | ログインボタン <!-- source: resources/views/auth/login.blade.php:15 --> |

### 引用ルール

- 表の「主要ファイル」カラムには行番号も含める
- 図（Mermaid）のノードには引用不要
- ファクトキャッシュの `source` フィールドが利用可能な場合はそれを使用する

## 3. 推測マークルール

### 基本ルール

- コードで直接確認できない記述には **【推測】** タグを付ける
- 組織的制約（チーム構成・デプロイ方針等）は【推測】対象
- 主要な技術選定の理由は【推測】対象（コードからは選定理由は判断できない）

### フロントエンド検出時の特例

フロントエンドが検出・分析済みの場合:
- 画面関連の記述に【推測】タグを付けない（実コンポーネントパスを記載）
- シーケンス図の `participant FE` に【推測】タグを付けず、`フロントエンド（{framework}）` と表記
- アーキテクチャ文書のフロントエンド技術名にも【推測】タグを付けない

**フロントエンドが検出されていない場合のみ【推測】タグを付ける。検出済みなら【推測】は禁止。**

### 特殊なコマンド

- **test-spec**: 【推測】の代わりに **【要記入】** を使用（期待結果でコードから判断できない項目）
- **user-guide / quick-ref**: 技術用語を使わないため、推測自体が発生しにくい。発生した場合は該当機能を記載しない

## 4. 数値整合チェック

出力完了前に以下の整合性を検証する:

- routes 件数 = API一覧の行数
- screens 件数 = 画面一覧の行数 = 画面詳細設計のセクション数
- tables 件数 = テーブル一覧の行数
- 図に登場するコンポーネント名 = テキストで説明されているコンポーネント名
- 用語が全セクションで統一されている（同じ概念に異なる名前を使わない）

## 4a. モジュール有効性検証ルール

### 原則

コードディレクトリが存在するだけではモジュール（機能）を有効とみなさない。
**実行時に有効であることをフレームワーク固有の設定・DB・ルーティングで確認**し、無効なモジュールは機能一覧・Tier分類に含めない。

### フレームワーク別の有効/無効判定方法

| フレームワーク | 有効性の判定方法 | 無効の代表例 |
|--------------|----------------|-------------|
| vtiger CRM | `vtiger_tab` テーブルの `presence` カラム（0=有効, 1=無効） | presence=1 のモジュール |
| Laravel | `config/app.php` の `providers` 配列 + ルーティング登録 | providers未登録・routes未定義 |
| Django | `settings.py` の `INSTALLED_APPS` + `urls.py` のルーティング | INSTALLED_APPS未登録 |
| Rails | `config/routes.rb` のルーティング + `config/application.rb` | routes未定義・engine未マウント |
| Next.js | `app/` or `pages/` ディレクトリのページファイル存在 | ページファイルなし・dynamic routeで除外 |
| WordPress | `wp_options.active_plugins` またはテーマの `functions.php` | active_plugins未登録 |
| 汎用PHP | エントリポイント（index.php等）からの include/require チェーン | require されていないモジュール |
| 汎用SPA | ルーター設定（vue-router, react-router等）への登録 | ルーター未登録のコンポーネント |

> **上記に該当しないフレームワークの場合**: 以下の汎用判定ルールを**順序通りに**確認する。最初にマッチした方法で判定すること。
>
> 1. **設定ファイル**（最優先）: `config/`, `src/config/`, `.env*` でモジュール有効化フラグ確認
>    - 例: `enabled: true/false`, `MODULES_ENABLED=["auth","api"]`, `active_plugins` リスト
> 2. **ルーティング定義**: ルーター/URL登録でエンドポイントが登録されているか
>    - 例: `router.addRoute()`, `route('/feature', ...)`, ページファイル存在
> 3. **メニュー/ナビゲーション定義**: UIメニュー、サイドバー、ナビゲーションバーからアクセス可能か
> 4. **DB設定テーブル**: システム管理テーブルの `enabled`, `status`, `presence` フラグ
> 5. **コード検索**（最終手段）: grep で `disabled`, `inactive`, `enabled: false` を検索
>
> **いずれの方法でも判定できない場合**:
> - `active: unknown` としてファクトに記録する
> - 本文に【推測】タグを付与し、「判定不能: モジュール管理メカニズムが特定できない」と注記する
> - 品質スコア（§7）のモジュール有効性軸は B 評価となる

### facts YAML への記録形式

既存の modules / components セクションに `active` フィールドを追加し、
無効モジュールは `disabled_modules` セクションにまとめる。

```yaml
facts:
  modules:  # または components, features 等
    - name: "モジュール名"
      active: true          # true / false / unknown
      active_source: "判定根拠（例: vtiger_tab.presence=0, config/app.php:45）"
      source: "(ファイル:行番号)"
  disabled_modules:         # active: false のモジュール一覧
    - name: "無効モジュール名"
      reason: "無効化の根拠（例: vtiger_tab.presence=1）"
      source: "(ファイル:行番号)"
```

### Tier 分類への適用

- **active: true** のモジュールのみ Tier 1 / Tier 2 / Tier 3 に分類する
- **active: false** のモジュールは `disabled_modules` → §11（技術的負債）に無効化モジュール一覧として記載する
- **active: unknown** のモジュールは【推測】タグ付きで Tier 分類に含め、判定不能の理由を注記する

### 漏れ防止チェック

出力完了前に以下を検証する:

- 有効モジュール全件 = Tier 1 + Tier 2 + Tier 3 の合計（1件も漏れ・重複がないこと）
- disabled_modules 全件が §11（技術的負債）に記載されていること

## 5. セルフレビューチェックリスト

出力前に必ず以下を実行する。C 評価の項目が1つでもあれば修正してから出力する。

| # | チェック項目 | 確認内容 |
|---|-----------|---------|
| 1 | **ファクト突合** | Step 0 のファクトに含まれない名称が本文に登場していないか |
| 2 | **件数一致** | ファクトの routes/screens/tables 件数と文書内の一覧件数が一致しているか |
| 3 | **引用検証** | `(file:line)` または `<!-- source: file:line -->` のパスがファクトに存在するか |
| 4 | **図テキスト整合** | Mermaid図のノード名が本文のコンポーネント一覧と一致するか |
| 5 | **用語統一** | 同じ概念に異なる名前を使っていないか |
| 6 | **推測マーク** | コードで直接確認できない記述に【推測】が付いているか。フロントエンド分析済みなら【推測】が残っていないか |
| 7 | **論理的深み** | 各セクションに「So What?」の答えがあるか。事実列挙だけになっていないか |
| 8 | **ビジネス接続** | 技術的記述がビジネスコンテキストと紐付いているか |
| 9 | **モジュール有効性** | Tier分類の全モジュールが active: true か。disabled_modules に漏れはないか |
| 10 | **修正ログ** | 上記チェックで修正した箇所を「修正: {箇所} {変更内容}」として記録 |

**注意**: コマンドによっては上記の一部が不要な場合がある（例: quick-ref は論理的深みチェック不要）。各コマンドファイルの「固有ルール」セクションで追加・除外を確認すること。

## 6. DB分析結果の活用ルール

`.cache/db-*.json` が存在する場合:

1. テーブル一覧は DB スキーマから取得した情報を正とする（コードのマイグレーションより優先）
2. カラム名・型・制約は DB スキーマの情報をそのまま転記する
3. ER図は DB スキーマのリレーション情報から生成する
4. マスタテーブルのレコード数・代表的な値を記載する
5. コードとDBの不一致（未使用カラム、マイグレーション未反映等）は明記する

## 7. 品質スコア基準

`scripts/score_quality.py` による自動評価の基準:

| 評価軸 | A | B | C |
|--------|---|---|---|
| ファクト根拠 | セクションの80%以上に (file:line) 引用あり | 50-79% | 50%未満 |
| 網羅性 | 期待ファイルの90%以上が存在 | 70-89% | 70%未満 |
| 図表充実度 | 10図以上 | 5-9図 | 5図未満 |
| 相互参照 | 内部リンク20件以上 | 10-19件 | 10件未満 |
| API完全性 | 20行以上 or OpenAPI仕様あり | 5-19行 | 5行未満 |
| テーブル完全性 | 10セクション以上 | 5-9セクション | 5セクション未満 |
| 画面網羅性 | 15セクション以上 | 5-14セクション | 5セクション未満 |
| 推測マーク | 0件 | 1-5件 | 6件以上 |
| モジュール有効性 | disabled_modules記載あり + Tier整合 | 一部チェック済み | 有効性検証なし |
| 形式品質 | 表紙・改訂履歴・ステータス全て存在 | 2/3 存在 | 1/3 以下 |

詳細な評価基準は `templates/quality-rubric.md` を参照。
