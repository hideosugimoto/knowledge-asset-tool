# /project:go

知識資産の生成をワンコマンドで完結させるウィザードです。
最初に必要な情報をすべて質問し、回答後は自動で最後まで実行します。

## Step 0: 環境チェック（初回のみ）

以下のツールの存在を確認し、未インストールの場合はインストールを提案する。

```bash
# 必須ツール
marp --version     || echo "未インストール: npm install -g @marp-team/marp-cli"
mmdc --version     || echo "未インストール: npm install -g @mermaid-js/mermaid-cli"
python3 -c "import yaml" || echo "未インストール: pip install pyyaml"
mkdocs --version   || echo "未インストール: pip install mkdocs-material"
```

未インストールのツールがある場合:
```
⚠️ 以下のツールが未インストールです:

  - marp (スライド生成に使用)
  - mmdc (Mermaid図のSVG変換に使用)

  一括インストールしますか？ (Y/n)
```

Y → 自動インストール実行。N → スキップ（該当機能は使えない旨を案内）。
全ツールがインストール済みの場合は何も表示せず Step 1 に進む。

## Step 1: ヒアリング

以下の質問をまとめてユーザーに聞いてください。番号付きで一度にすべて質問し、回答を待ってください。
途中で追加の質問はしないでください。

```
知識資産を生成します。以下の質問に回答してください。

1. 対象パス（分析したいコードのパス）
   例: ./src/auth, /Users/.../project

2. 名前（出力ファイル名に使います。英数字・ハイフン推奨）
   - a)〜e) の場合: 機能名（例: user-authentication, payment-flow）
   - f) g) の場合: システム名（例: my-app, my-app）

3. 出力モード（番号で選択、複数可）
   a) 標準（構造・詳細・レビュー）
   b) 完全版（全セクション）
   c) RAG用（最小・高速）
   d) 共有用（レベル別説明）
   e) スライド（20-30枚の全機能網羅版）
   f) 完全マニュアル（全画面・全機能・全API・画面遷移図・画面詳細設計）
   g) マニュアルベーススライド（生成済みマニュアルからSlidedocs形式で作成）
   h) AI向けドキュメント（{名前}-llms.txt + {名前}-AGENTS.md）
   j) ドキュメントサイト（MkDocs Material で検索付きWebサイト生成）
   k) 操作ガイド（エンドユーザー向け。技術用語なし・全画面ステップバイステップ手順書）
   l) クイックリファレンス（A4 1枚の早見表。印刷してデスクに貼れる）
   m) テスト仕様書（コードからテスト項目を自動抽出。期待結果は【要記入】プレースホルダ）
   n) テスト仕様書 + 運用設計書 + セキュリティマトリクス（検収パッケージ）
   i) 全部（b + e + f + g + h + j + k + l + m + n を実行。全成果物を一括生成。b にRAG出力が含まれるため c は省略）

4. e), g), h) のいずれかを選んだ場合のみ回答：
   - 対象読者: エンジニア / 営業 / 初心者（デフォルト: エンジニア）
   - 出力形式: html / pdf / pptx（デフォルト: html）

5. 検収成果物モード（任意）
   検収成果物として提出する場合は以下を入力してください（不要なら「スキップ」）:
   - プロジェクト名（正式名称）
   - 文書番号プレフィックス（例: SYS-2026）
   - 作成者名
   - 確認者名
   - 承認者名
   - 機密区分（社外秘 / 関係者限り / なし）
```

**保存先は常に `./docs` 固定。** ユーザーには聞かない。
プロジェクト名付きファイル名により、複数プロジェクトが同じ `./docs` に安全に共存する。

## Step 1.3: フロントエンド自動検出

対象パスに対して `python3 scripts/detect_frontend.py --source-dir {対象パス} --human` を実行。

**フロントエンドが検出された場合:**
```
フロントエンドプロジェクトが検出されました。

  検出されたフロントエンド:
    1. {framework} ({path}) -- {page_count}画面, {store_count}ストア

```

ユーザーが Y (またはデフォルト) → 対象パスにフロントエンドパスを追加して分析。
N → バックエンドのみで分析（画面関連は【推測】タグ付き）。

**検出されなかった場合:**
何も表示せず次のステップへ進む。

## Step 1.5: DB連携確認

**対象パスの回答を受け取ったら、DB連携の確認を行う。**

### 1.5-A. DB接続設定の自動検出

対象パスに対して `python3 scripts/scan_database.py --source-dir {対象パス}` を実行し、
検出されたDB接続設定とマイグレーションディレクトリを取得する。

### 1.5-B. 検出結果の提示と確認

検出結果をユーザーに提示し、確認を求める。

**DB接続設定が検出された場合:**
```
データベース接続設定が検出されました。

  検出されたDB接続:
    1. [{種別}] {DB名} (ホスト: {host}:{port}) — 設定元: {ファイル名}
    2. [{種別}] {DB名} (ホスト: {host}:{port}) — 設定元: {ファイル名}
    ...

  検出されたマイグレーション:
    - {ディレクトリパス} ({N}ファイル)

以下を確認してください:
  Q1. 上記以外に使用しているDBはありますか？（あればDB種別・名前・用途を教えてください）
      ※ 独自の接続設定ファイル経由でDBに繋いでいる場合もあるので確認すること
  Q2. 各DBの用途を教えてください（例: メインDB、ログDB、キャッシュ等）
  Q3. 分析対象から外すDBはありますか？（ログDB、キャッシュ等は通常除外を推奨）
  Q4. DB分析の方法を選んでください（複数DBがある場合はDB毎に指定可）:
      (a) ライブDB接続（SQLを実行できるコマンドを教えてください）
      (b) SQLダンプファイルを指定（パスを教えてください）
      (c) DB分析をスキップ（コード分析のみ）
```

**(a) ライブDB接続を選択した場合の追加質問:**
```
DBにSQLを実行するためのコマンドを教えてください。
SQLはこのコマンドの標準入力にパイプして実行します。

例:
  - mysql -u root -N mydb
  - docker exec -i mysql_container mysql -u root mydb
  - ssh app-server 'mysql -u app -pPASS mydb'
  - kubectl exec -i pod/mysql -- mysql -u root mydb
  - psql -U postgres -t mydb
  - その他、プロジェクト独自の接続方法があればそのコマンド

DB種別も教えてください: mysql / postgresql
```

**DB接続設定が検出されなかった場合:**
```
データベース接続設定は検出されませんでした。

このプロジェクトはDBと連携していますか？
  (a) はい → DB種別・名前・用途を教えてください
  (b) いいえ → DB分析をスキップしてコード分析のみ実行します
```

### 1.5-C. DB分析の実行

ユーザーの回答に基づいて、分析対象の各DBに対してスキーマ情報を取得する:

- **(a) ライブDB接続**: `python3 scripts/scan_database.py --sql-command "{ユーザー指定のコマンド}" --db-type {mysql|postgresql} --output .cache/db-{DB名}.json`
- **(b) SQLダンプ**: `python3 scripts/scan_database.py --dump-file {パス} --output .cache/db-{DB名}.json`
- **(c) スキップ**: DB分析なしで Step 2 へ進む

取得した結果は `.cache/db-{DB名}.json` に保存し、後続の分析ステップで参照する。

**⚠️ ライブDB接続では読み取りクエリ（SELECT）のみを実行する。書き込みは一切行わない。**

### 1.5-D. DB構成の最終確認

分析結果をまとめてユーザーに提示する:
```
DB構成を以下のように理解しました。

  分析対象:
    1. {DB名} ({種別}) — {用途} — テーブル数: {N} ✅ 分析対象
    2. {DB名} ({種別}) — {用途}                  ❌ 対象外

  検出されたマスタテーブル: {N}件
  有効/無効フラグ付きテーブル: {N}件
  設定テーブル: {N}件
  ラベル/名称カラムを持つテーブル: {N}件

問題なければ「OK」と入力してください。修正があれば指摘してください。
```

**⚠️ ユーザーの「OK」を受け取るまで分析を開始しない。**

## Step 3: 分析実行

**ファクトキャッシュ**:
各コマンドの Step 0（ファクト収集）に入る前に `python3 scripts/cache_analysis.py --action check-facts --source-dir {対象パス} --name {名前}` を実行。
キャッシュが有効な場合は `.cache/facts-{名前}.yaml` から読み込んでファクト収集をスキップし、
「キャッシュ済みファクトを使用（{生成日時}、コミット {hash}）」と表示する。
キャッシュが無効（ソース変更あり）または存在しない場合は通常通りファクト収集を実行し、
完了後に `python3 scripts/cache_analysis.py --action save-facts --source-dir {対象パス} --name {名前}` でキャッシュに保存する。

**フロントエンド情報の引き渡し（Step 1.3 で検出された場合）:**
Step 1.3 でフロントエンドが検出され、ユーザーが分析対象に含めることを承認した場合、
以下の全コマンド実行時にフロントエンドパスも分析対象に含めること：
- ファクト収集（Step 0）でフロントエンドの pages/, store/, components/ も読む
- モジュール有効性チェック: `quality-rules.md` §4a に従い各モジュールの active を判定する（コードの存在だけでモジュールを列挙しない）
- 画面関連の記述（第2章, 第6章, features/*.md の「画面」節）は実際のVueテンプレートから復元し、【推測】タグは付けない
- シーケンス図の `participant FE` には【推測】タグを付けず、`フロントエンド（{framework}）` と表記する
- architecture文書のフロントエンド技術名にも【推測】タグを付けない（検出済みの情報を使用）

**⚠️ フロントエンドが検出されていない場合のみ【推測】タグを付ける。検出済みなら【推測】は禁止。**

**検収成果物モードが有効な場合の表紙挿入ルール:**
検収モードが有効な場合、各生成ドキュメントの先頭に `templates/cover-page.md` の内容を挿入する。
プレースホルダーは Step 1 で収集した情報で埋める。文書番号（DOC_ID）は以下のルールで自動採番する:
- `architecture/*.md` → `{prefix}-AD-001`
- `manual/*/00-index.md` → `{prefix}-DD-001`（詳細設計書）
- `manual/*/04-api-reference.md` → `{prefix}-API-001`
- `manual/*/05-data-model.md` → `{prefix}-DB-001`
- `manual/*/06-screen-specs.md` → `{prefix}-SD-001`（画面設計書）
VERSION は `1.0`、STATUS は `DRAFT`。改訂履歴の初版エントリを自動挿入する。

## Step 4: ファイル保存（スライド以外）

出力モードが a)〜d), f), h), k), l), m), i) の場合、分析結果を `--- FILE: docs/... ---` 形式で出力したあと、
自動で以下のコマンドを実行してファイルに保存してください。

```bash
python scripts/save_output.py \
  --name {機能名} \
  --output-dir ./docs \
  --force
```

ただし save_output.py はクリップボードまたは stdin からテキストを受け取るため、
分析結果を直接ファイルに書き出す方法で保存してください：
- 各 `--- FILE: パス ---` セクションの内容を、`./docs` 配下にそのまま書き出す
- ディレクトリが存在しない場合は自動作成する
- `docs/` プレフィックスは除去してから `./docs` と結合する

## Step 4.5: プレースホルダー解消パス（全モード共通・必須）

成果物が生成されたら、全ファイルから `【推測】` `【要記入】` `要確認` を洗い出し、可能な限り解消する。

### 4.5-A. プレースホルダーの検出

```bash
grep -rn '【推測】\|【要記入】\|要確認' docs/
```

### 4.5-B. 各項目の分類と対応

検出された各項目を以下の3カテゴリに分類し、それぞれ対応する:

1. **ソースコードまたはDBで確認可能** → 実際に読んで正確な値に置換する
   - 例: ライブラリのバージョン → package.json / composer.json / 実ファイルから特定
   - 例: テーブル名・カラム名 → マイグレーションファイルやDB分析結果から特定
   - 例: API エンドポイントの詳細 → ルーティング定義から特定

2. **実環境でしか確認不可能**（デプロイ構成・SLA・運用手順等） →
   推測を断言から「〜と推測される（実環境での確認が必要）」の表現に修正し、【推測】タグを除去する

3. **根拠がなく削除すべきもの** → 該当記述を削除する

### 4.5-C. 検証

```bash
grep -rn '【推測】\|【要記入】\|要確認' docs/
```

ゼロ件になることを確認する。ゼロ件にならない場合は、残存項目を完了報告の「未解決の品質問題」に記載する。

## Step 5: 目次生成（全モード共通・必須）

成果物が1つでも生成されたら、必ず以下を実行してください。

### 5-A. プロジェクト固有の目次ファイル

**⚠️ 目次ファイルは `./docs/{名前}-index.md` として生成する。`./docs/index.md` を上書きしてはならない。**
複数プロジェクトの成果物が同じ `./docs` に共存するため、`index.md` を直接使うと前回の目次が消える。

1. `templates/index-template.md` を読み込む
2. 以下のプレースホルダーを置換して `./docs/{名前}-index.md` を生成：
   - `{NAME}` → 機能名/システム名
   - `{DATE}` → 今日の日付
   - `{TARGET_PATH}` → 対象パス
   - `{FEATURES_TABLE}` → `features/` 配下のファイル一覧をテーブル形式で生成
   - `{DIAGRAMS_TABLE}` → `diagrams/` 配下の {NAME}-*.mmd / .svg ファイル一覧をテーブル形式で生成
3. **存在しないファイルへのリンクは除外する**（生成されなかった成果物のセクションは丸ごと削除）
4. 目次ファイルに「共有方法」セクションを含める
5. 鮮度バッジを挿入する:
   ```bash
   python3 scripts/freshness_badge.py --source-dir {対象パス} --name {名前} --docs-dir ./docs --insert
   ```

### 5-B. ポータル（index.md）への追記

`./docs/index.md` がまだ存在しない場合は、以下のヘッダーで新規作成する：
```markdown
# knowledge-asset-tool ドキュメントポータル

> ツール: knowledge-asset-tool

---

## プロジェクト一覧

| プロジェクト | 説明 | 目次 |
|------------|------|------|
```

既に存在する場合は、プロジェクト一覧テーブルに **行を追記** する（既存行は削除しない）。
同名プロジェクトの行が既にあれば、その行を更新する。

**⚠️ 冪等性のためマーカーコメントを使用する:**
各プロジェクト行の前後にHTMLコメントを付与し、更新時の検出に使う：
```markdown
<!-- PROJECT:{名前} -->
| **{名前}** | {説明} | [{名前}-index.md]({名前}-index.md) |
<!-- /PROJECT:{名前} -->
```
同名マーカーが既にあれば、その行を置換する。なければテーブル末尾に追記する。

## Step 5.5: 品質チェック（全モード共通・必須）

成果物が生成されたら、以下の品質チェックスクリプトを自動実行してください。

```bash
# 1. リンク整合性チェック — 壊れたリンクを検出
python3 scripts/check_links.py --docs-dir ./docs

# 2. 用語一貫性チェック — 表記ゆれを検出
python3 scripts/check_consistency.py --docs-dir ./docs

# 3. スライド溢れチェック — 画像切れリスクを検出（スライド生成時のみ）
python3 scripts/check_slide_overflow.py --docs-dir ./docs

# 4. 図の自動分割 — 超過図を閾値内に分割
python3 scripts/split_diagram.py --docs-dir ./docs --name {名前} --auto
```

- エラーが検出された場合は**自動修正を試みる**（リンク先の修正、用語の統一、Mermaid ノード数の削減）
- 修正後に再度チェックを実行し、エラーが0になるまで繰り返す（最大3回）
- 修正不能なエラーは完了報告の「未解決の品質問題」セクションに記載する

```bash
# 5. 品質スコア自動評価 — ルーブリックに基づくスコアリング
python3 scripts/score_quality.py --docs-dir ./docs --name {名前}
```

- 品質スコアレポートを完了報告に含める
- 総合スコアが B 未満の場合は、改善提案に基づいて自動修正を試みる（最大2回）
- 修正後に再スコアリングし、改善結果を完了報告に記載する

```bash
# 6. Tier 3 網羅性チェック — ファクト上のモジュール件数と本文記述の一致を検証
# ファクトキャッシュ .cache/facts-{名前}.yaml のモジュール/機能リストと、
# 生成ドキュメント内に記述されたモジュール件数を比較する。
# 不足がある場合は templates/priority-analysis.md の最低基準に従って追記する。
```

## Step 6: 共有用サイトビルド（全モード共通・必須）

`mkdocs` がインストール済みの場合、以下を実行してフォルダコピーで共有可能なHTMLサイトを生成：

```bash
mkdocs build
```

`site/` ディレクトリに自己完結型のHTMLサイトが生成される。
このフォルダをそのまま共有相手に渡せば、ブラウザで `site/index.html` を開くだけで閲覧できる。

`mkdocs` 未インストールの場合は `pip3 install --break-system-packages mkdocs-material` でインストールする。

## Step 6.3: スクリーンショット取得（到達可能時は自動提案）

分析対象のアプリケーションが稼働中の場合、画面キャプチャを取得してドキュメントに挿入する。

### 6.3-A. 到達可能性チェック（自動実行）

Step 0 のファクト収集で取得した screens/routes データからアプリの base URL を推定し、
到達可能性を自動チェックする:

```bash
python3 scripts/capture_screenshots.py --base-url {推定URL} --routes /dev/null --output-dir /tmp --check-only
```

### 6.3-B. 到達可能な場合（自動提案）

```
アプリケーション ({URL}) が稼働中です。スクリーンショットを取得して操作ガイドに挿入できます。

  (a) はい（デフォルト） → スクリーンショットを取得
  (b) いいえ → スキップ
```

**(a) または無回答の場合、認証の有無を確認する:**

ファクト収集で認証機能（ログイン画面・認証ミドルウェア等）が検出された場合、以下を質問する:
```
ログイン認証が検出されました。スクリーンショット撮影のためにログイン情報を教えてください。

  ログインURL: （例: {URL}/login — 検出結果から自動入力）
  ID（メールアドレス等）:
  パスワード:

  ※ この情報はスクリーンショット撮影にのみ使用し、生成ドキュメントには一切記載しません。
  ※ 認証不要の場合は「不要」と入力してください。
```

認証機能が検出されなかった場合は認証なしで撮影を開始する。

**撮影実行:**
1. Step 0 のファクト収集の screens/routes データから routes.json を自動生成し `/tmp/{名前}-routes.json` に保存
2. 認証情報がある場合は `--login-url` `--email` `--password` オプション付きで実行:
   ```bash
   python3 scripts/capture_screenshots.py --base-url {URL} --routes /tmp/{名前}-routes.json --output-dir ./docs/screenshots/{名前}/ --login-url {ログインURL} --email {ID} --password {パスワード}
   ```
   認証不要の場合はオプションなしで実行
3. 取得したスクリーンショットを以下のファイルに自動挿入:
   - 操作ガイド（09-user-guide.md）の各章: `![{画面名}](../screenshots/{名前}/{画面名}.png)`
   - 画面設計書（06-screen-specs.md）のカスタム画面セクション: `![{画面名}](../screenshots/{名前}/{画面名}.png)`

**⚠️ セキュリティルール: ユーザーから受け取ったID・パスワード等の認証情報は、生成ドキュメント・ログ・コメント・コミットメッセージ等の一切の成果物に含めてはならない。スクリプト実行時の引数としてのみ使用する。**

### 6.3-C. 到達不可能な場合

スクリーンショット取得をスキップし、以下の2つを実行する:
1. 操作ガイド（09-user-guide.md）の各章末に以下のコメントを挿入:
   `<!-- TODO: アプリ起動後に capture_screenshots.py でスクリーンショットを追加 -->`
2. 完了報告の「未解決の品質問題」に以下を記載:
   `スクリーンショット未取得（アプリ到達不可）: capture_screenshots.py で後から追加可能`

## Step 6.7: スライドアセット整理（スライド生成時のみ）

スライドが生成された場合、画像パスを修正して自己完結型にする:

1. 生成したスライド Markdown 内の全画像パスを走査する
2. `docs/` 外（`/tmp/` 等）を参照している画像ファイルを `docs/slides/{名前}/images/` にコピーする
3. スライド Markdown 内のパスを相対パス `images/{filename}` に書き換える

**⚠️ このステップは Step 5.5（品質チェック）より後、Step 6.8 より前に実行する。**

## Step 6.8: FINAL 昇格判定（全モード共通・必須）

以下の全条件を満たす場合、全ファイルの `ステータス: DRAFT` を `ステータス: FINAL` に置換する。
1条件でも満たさない場合は DRAFT のまま残し、完了報告の「未解決の品質問題」に
FINAL にならなかった理由を記載する。

**昇格条件:**
- [ ] `grep -rn '【推測】' docs/` の結果が 0 件
- [ ] `grep -rn '【要記入】' docs/` の結果が 0 件
- [ ] `python3 scripts/check_links.py --docs-dir ./docs` でリンク切れ 0 件
- [ ] `python3 scripts/score_quality.py --docs-dir ./docs --name {名前}` の総合スコアが B+ 以上
- [ ] 操作ガイドにスクリーンショットが含まれている（アプリ到達可能だった場合のみ）

**昇格実行:**
条件を全て満たした場合、以下を実行:
```bash
# 全ドキュメントの DRAFT → FINAL 置換
grep -rl 'ステータス: DRAFT' docs/ | while read f; do
  sed -i '' 's/ステータス: DRAFT/ステータス: FINAL/g' "$f"
done
```

## 重要ルール

- Step 1 の質問はまとめて一度だけ行う。追加質問はしない
- ユーザーの「OK」を受け取るまで分析を開始しない
- 分析中に確認を求めない。すべて自動で進める
- エラーが発生した場合のみ報告する
- 分析には .claude/commands/ の各コマンドファイルの指示をそのまま適用する
- **Step 5（目次）と Step 6（サイトビルド）は全モードで必ず実行する**
- **サブエージェント委譲時の品質ルール伝播**: Step 3 の分析を Agent ツール（サブエージェント）に委譲する場合、以下を必ず遵守する：
  1. サブエージェントへの指示文に `templates/quality-rules.md を先に読み、そのルールに従うこと` を含める
  2. サブエージェントへの指示文に、実行するコマンドファイルのパス（例: `.claude/commands/manual.md`）を含め、「このファイルの実行内容セクションに従うこと」と指示する
  3. ファクトキャッシュ `.cache/facts-{名前}.yaml` のパスを指示文に含め、「各エントリの source フィールドをソース引用に使うこと」と指示する
  4. サブエージェントに**テキスト要約でファクトを渡さない**。必ずファイルパスを渡し、サブエージェント自身に読ませる
