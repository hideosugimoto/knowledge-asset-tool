# 重要度ベース分析ガイド

全機能を同じ深さで書くのではなく、ビジネスインパクトに基づいて深掘り度を変えるためのガイド。
一流SIの成果物は重要な箇所に厚く、補助的な箇所に薄くメリハリがある。

## 1. Tier 分類基準

### Tier 1: コア機能（詳細記述）
ビジネスの主要なバリューストリームを構成する機能。これが止まるとビジネスが止まる。

**判定指標**:
- ビジネスコンテキストの `purpose` に直結する機能
- コントローラーのメソッド数が多い（5+）
- 関連テーブル数が多い（3+）
- 関連画面数が多い（2+）
- フロントエンドコンポーネントが複雑（子コンポーネント3+）
- 独自のビジネスロジック・計算ロジックがある

### Tier 2: 重要機能（標準記述）
コア機能を支える機能。なくても業務は回るが、品質や効率が大幅に低下する。

**判定指標**:
- 認証・認可（全機能のゲートキーパー）
- マスタデータ管理（コア機能のデータ基盤）
- 設定管理（システム動作の制御）
- レポート・帳票（業務の出口）

### Tier 3: 補助機能（サマリー記述）
あると便利だが、なくてもビジネスは回る。標準的な CRUD パターンで実装されている。

**判定指標**:
- コントローラーのメソッド数が少ない（4以下で全て標準CRUD）
- 独自のビジネスロジックがない
- 画面が1つだけの単純な管理機能
- エクスポート、通知設定、ログ表示、プロフィール編集等

## 2. Tier 分類の目安配分

| Tier | 全機能に占める割合 | 記述の深さ |
|------|:----------------:|-----------|
| Tier 1 | 20-30% | テンプレート全項目 + 追加深掘り |
| Tier 2 | 40-50% | テンプレート全項目（標準） |
| Tier 3 | 20-30% | 概要 + コンポーネント表 + API一覧 |

**注意**: Tier 1 が 40% を超える場合は基準が甘い。Tier 3 がゼロの場合は基準が厳しすぎる。
**前提**: 上記パーセンテージは active: true のモジュールのみが対象。disabled_modules（active: false）は Tier 分類対象外とし、技術的負債セクションに記載する（`quality-rules.md` §4a 参照）。

## 3. Tier 別の記述テンプレート

### Tier 1: 詳細記述

標準テンプレートの全項目に加えて以下を含む：

- **ビジネスルール詳細**: 計算式、条件分岐、ステータス遷移を網羅的に記述
- **エッジケース**: 境界値、例外パターン、同時操作時の挙動
- **パフォーマンス考慮点**: クエリの効率、N+1問題の有無、キャッシュ戦略
- **データフローの全体図**: リクエストからレスポンスまでの完全なフロー（シーケンス図に加えて、データ変換の詳細）
- **テスト観点**: テストすべきケースの推奨リスト

### Tier 2: 標準記述

現行のテンプレート全項目をそのまま適用する（これが今までの「全機能共通」の深さ）。

### Tier 3: サマリー記述 — 最低基準

Tier 3 であっても以下は必須（名前だけの1行は禁止。最低4行を確保する）:

- **日本語名**: 言語ファイル・設定ファイルから取得（取得できない場合はクラス名から推定し注記）
- **機能説明**: 1-2文の説明（メインクラスまたはエントリポイントを読んで判断）
- **メインテーブル情報**: テーブル/コレクションのカラム数・行数（DB分析結果がある場合）
- **カスタムロジックの有無**: 保存フック・バリデーション・イベントリスナーの存在チェック結果
  （「カスタムロジックなし。標準CRUDパターンのみ」でもよいが、確認した事実として記載する）

以下の項目は任意（あれば記載）:
- **関連コンポーネント表**: Controller, Repository, Model, Store
- **APIエンドポイント一覧**: メソッド + パス + 説明
- **共通パターン参照**: 「共通CRUDパターンと同一」等

省略してよい項目：
- シーケンス図（共通CRUDパターンと同一の場合）
- ビジネスルール（独自のものがない場合）
- 注意点（特にない場合）

### Tier 3 網羅性の検証ルール

モジュール/機能一覧の章の生成時に、以下の検証を行うこと:

1. ファクト収集で列挙した全モジュール/全機能のリストを参照する
2. 本文に記述されたモジュール件数がファクト上のモジュール件数と一致するか検証する
3. 不一致がある場合は不足しているモジュールを追記し、最低基準を満たす記述を追加する
4. DB分析結果がある場合は、全モジュールのレコード件数を取得してモジュール一覧とデータモデルの章に反映する

## 4. コマンド別の適用範囲

| コマンド | Tier適用 | 補足 |
|---------|:-------:|------|
| analyze | Yes | §5 ビルディングブロック、§6 ランタイムでメリハリ |
| analyze-full | Yes | 全セクションで Tier を意識。C-1 プロ向けで特に有効 |
| manual | Yes（ただし制限あり） | 第3章の機能カタログで適用。**第6章の画面詳細設計は全画面必須のため Tier 3 でもフォーム仕様・テーブル仕様は省略しない** |
| analyze-slide | Yes | Tier 1 に多くのスライドを割く |
| analyze-share | Yes | Tier 1 を中心に説明 |
| analyze-rag | **No** | 均等チャンクが前提のため不適用 |
| user-guide | **No** | 全画面の操作手順を網羅する前提のため不適用 |
| quick-ref | **No** | A4 1枚の制約が優先のため不適用 |

## 5. 出力フォーマット

```yaml
priority_map:
  tier1:
    - feature: "(機能名)"
      reason: "(コア判定の理由)"
      indicators:
        controllers: N
        tables: N
        screens: N
        api_endpoints: N
        has_business_logic: true
  tier2:
    - feature: "(機能名)"
      reason: "(重要判定の理由)"
  tier3:
    - feature: "(機能名)"
      reason: "(補助判定の理由)"
  disabled:
    # active: false のモジュール（Tier分類対象外）
    - feature: "(無効モジュール名)"
      reason: "(無効化の根拠)"
  distribution:
    tier1_pct: N%
    tier2_pct: N%
    tier3_pct: N%
```

## 6. 悪い例と良い例

### 悪い例: 全機能均等

12機能を全て同じ深さで書いた結果、コア機能の「売上集計」が他の「プロフィール編集」と同じ分量になっている。読者は何が重要かわからない。

### 良い例: Tier でメリハリ

```
Tier 1（30%の機能に50%の分量）:
  - 売上管理: 詳細なビジネスルール、計算式、エッジケース、パフォーマンス分析
  - 請求処理: ステータス遷移図、外部API連携の詳細、エラーリカバリー

Tier 2（40%の機能に35%の分量）:
  - 認証・権限: 標準テンプレート全項目
  - 顧客管理: 標準テンプレート全項目
  - マスタ管理: 標準テンプレート全項目

Tier 3（30%の機能に15%の分量）:
  - 通知設定: 概要 + API一覧（共通CRUDパターン参照）
  - プロフィール編集: 概要 + API一覧（共通CRUDパターン参照）
  - ログ表示: 概要 + API一覧
```
