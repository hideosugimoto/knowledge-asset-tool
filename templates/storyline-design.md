# ストーリーライン設計ガイド

テンプレート埋めではなく「読者に何を伝えたいか」を先に設計してから書くためのガイド。
ピラミッドプリンシプル（結論先行 → 根拠 → 詳細）に基づく。

## 1. ピラミッドプリンシプルの適用

### 基本構造

```
メインメッセージ（ドキュメント全体の結論）
├── サブメッセージ1（セクション/章の結論）
│   ├── 根拠1-1（ファクトに基づく裏付け）
│   └── 根拠1-2
├── サブメッセージ2
│   ├── 根拠2-1
│   └── 根拠2-2
└── サブメッセージ3
    └── ...
```

### 適用ルール

1. **結論先行**: 各セクション・各章の冒頭に「このセクションの結論」を1-2文で提示する
2. **帰納法で根拠**: 結論を支える3-5個の根拠を箇条書きで示す。根拠は全てファクト（Step 0）に基づく
3. **詳細は根拠の下**: 各根拠の裏付けとなるコード引用・データを、根拠の後に配置する
4. **空中戦禁止**: テンプレートを埋める前に、まずメインメッセージを決める。メッセージが決まらないまま書き始めない

## 2. メインメッセージの設計テンプレート

### analyze / analyze-full

```
このシステムは {ビジネスコンテキストの目的} を
{アーキテクチャパターン} で実現しており、
{主要な強み（1-2個）} がある一方、
{主要なリスク（1-2個）} に注意が必要である。
```

例:
> このシステムは月次売上集計の自動化を Laravel + Vue.js の SPA 構成で実現しており、CRUD操作の統一されたパターンと柔軟なロール管理が強みである一方、テストカバレッジの不足と一部のN+1クエリに注意が必要である。

### manual

```
このマニュアルでは {システム名} の全 {N} 画面・{N} API・{N} テーブルの仕様を網羅し、
開発・保守に必要な全情報を提供する。
特に {コア機能} の設計が中心であり、{注意すべき設計判断} を理解することが重要である。
```

### analyze-slide（エンジニア向け）

```
{システム名} は {技術スタック} による {アーキテクチャパターン} で、
{N} 個の主要機能を {M} 画面で提供する。
コードリーディングのエントリーポイントは {起点ファイル} である。
```

### analyze-slide / analyze-share（営業向け）

```
{システム名} は {ビジネスの課題} を解決し、
{定量的効果} をもたらすシステムである。
{N} 個の主要機能で {業務フロー} を効率化する。
```

### user-guide

```
この操作ガイドに従えば、{システム名} の全 {N} 画面の操作を迷わず実行できます。
日常業務で最も使う機能は {コア機能} で、{主な操作の流れ} が基本です。
```

### quick-ref

```
{システム名} のよく使う操作 TOP10 と全 {N} 画面の早見表です。
```

## 3. セクション別サブメッセージの設計

各セクションを書く前に、そのセクションのサブメッセージを1文で定義する。

### 例: manual の第3章（機能カタログ）

```yaml
storyline:
  main_message: "このシステムの全N機能の仕様を網羅し、開発・保守に必要な全情報を提供する"
  sub_messages:
    - section: "第1章"
      message: "このシステムはLaravel+Vue.jsによる業務システムで、5つのサービスから構成される"
    - section: "第2章"
      message: "全18画面は認証状態で2つのフローに分かれ、サイドバーから直接アクセスできる"
    - section: "第3章"
      message: "全12機能はデータ入力・処理・出力の3層に分類され、売上管理がコア機能である"
    - section: "第4章"
      message: "REST APIは45エンドポイントで、JWT認証+ロールベースの3段階権限で保護されている"
    - section: "第5章"
      message: "35テーブルはマスタ系・トランザクション系・システム系に分類され、正規化は適切に行われている"
```

**注意**: ストーリーラインに含める機能（サブメッセージの根拠）は active: true のモジュールのみ。disabled_modules はメインストーリーに登場させない（`quality-rules.md` §4a 参照）。

## 4. 悪い例と良い例

### 悪い例（テンプレート埋め）

```markdown
## §1. はじめに

### システムの目的
売上管理を行うシステムです。

### 主要な機能要件
- ユーザー管理
- 売上管理
- 請求管理
- レポート出力
- 設定管理

### 品質目標
- 性能
- 可用性
- 保守性
```

### 良い例（ストーリーライン設計済み）

```markdown
## §1. はじめに

このシステムは、複数拠点の売上データを一元管理し、月次請求処理を自動化することで、
経理部門の手作業を年間推定500時間削減することを目的とした業務システムである。

### 主要な機能要件（MECE: データ入力→処理→出力の3層）

**データ入力層**（日次業務の入口）:
- 売上データの登録・修正（SalesController:12-45）
- 配送情報の管理（DeliveryController:8-30）

**処理・集計層**（ビジネスロジックの中核）:
- 月次売上集計と自動計算（BillingService:23-89）
- 請求書生成（InvoiceService:15-67）

**出力・連携層**（業務の出口）:
- PDF帳票出力（PdfExportService:10-45）
- 外部会計システムへのデータ連携（AccountingApiClient:5-30）

### 品質目標（ビジネス要件から導出）

| 品質属性 | 目標 | 根拠 |
|---------|------|------|
| 応答性能 | ダッシュボード3秒以内 | 日次確認で毎日使用するため（routes/web.php:12） |
| データ整合性 | トランザクション保証 | 金額を扱うため不整合は許容不可（BillingService:45） |
| 可用性 | 月末の集中アクセスに耐える | 月次締め処理が業務のクリティカルパス |
```

## 5. ストーリーライン設計の出力フォーマット

```yaml
storyline:
  main_message: "(ドキュメント全体の結論)"
  target_reader: "(想定読者: エンジニア/営業/初心者/混合)"
  sub_messages:
    - section: "(セクション/章名)"
      message: "(このセクションの結論)"
      required_facts: [routes, controllers, ...]  # Step 0で重点収集する対象
  emphasis:
    - "(特に強調すべきポイント)"
  narrative_flow: "(問題提起→解決策→詳細→まとめ / 結論先行→根拠→詳細 等)"
```
