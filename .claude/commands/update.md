# /project:update

コード変更後にドキュメントの差分更新を実行します。
前回の生成時点からの変更を分析し、影響のある章だけを再生成します。

## 引数

$ARGUMENTS をスペース区切りで以下のように解釈してください：
- 第1引数: 対象パス
- 第2引数: 名前（既存の生成物の名前）

## 実行手順

### Step 1: 変更差分の検出

`python3 scripts/diff_update.py --source-dir {対象パス} --name {名前} --docs-dir ./docs` を実行し、
前回生成時からの変更ファイルと影響を受けるドキュメント章を特定する。

### Step 2: 影響範囲の表示

検出された変更を表示し、ユーザーに確認する。

```
前回生成時（{コミットハッシュ}）からの変更:

  変更ファイル: {N}件
  影響を受ける章:
    - 第3章 機能カタログ（{変更理由}）
    - 第4章 APIリファレンス（{変更理由}）
    - 第6章 画面詳細設計（{変更理由}）

  変更なしの章（スキップ）:
    - 第1章, 第2章, 第5章, 第7章, 第8章

再生成しますか？ (Y/n)
```

### Step 3: 影響章の再生成

影響を受ける章のみ、対応するコマンドファイルの指示に従って再生成する。
- ルート/コントローラー変更 → 第3章, 第4章
- モデル/マイグレーション変更 → 第5章
- ビュー/コンポーネント変更 → 第2章, 第6章
- 設定/ミドルウェア変更 → 第1章, 第8章

再生成時は既存ファイルを上書きする。変更なしの章はスキップ。

### Step 4: 品質チェック・サイトリビルド

go.md の Step 5.5（品質チェック）と Step 6（サイトビルド）を実行。

### Step 5: 完了報告

```
差分更新完了。

  前回: {コミットハッシュ} ({日時})
  今回: {コミットハッシュ} ({日時})

  再生成した章: {N}章
    - {章名}: {変更理由}
    ...
  スキップした章: {N}章

  共有用サイト: site/ (リビルド済み)
```

## 重要ルール
- 差分がない場合は「変更なし」と報告して終了
- 影響判定が不明確な場合は安全側に倒して再生成
- 再生成する章は go.md と同じ品質基準を適用
