# /project:analyze-slide

コードを分析してスライド資料を自動生成します。
HTML・PDF・PPTXの好きな形式で出力できます。
Google スライドへのインポートも可能です。

## 引数

$ARGUMENTS をスペース区切りで以下のように解釈してください：
- 第1引数: 対象パス
- 第2引数: 機能名（ファイル名に使用）
- 第3引数（任意）: 対象読者（エンジニア / 営業 / 初心者、省略時: エンジニア）
- 第4引数（任意）: 形式（html / pdf / pptx、省略時: html）

```
例: /project:analyze-slide ./src/auth user-authentication エンジニア pdf
例: /project:analyze-slide ./src/auth user-authentication 営業 pptx
例: /project:analyze-slide ./src/auth user-authentication 初心者 html
```

以降、第1引数を「対象パス」、第2引数を「機能名」、第3引数を「対象読者」、第4引数を「形式」として参照します。

## 実行手順

### Step 1: ツールの確認

まず `marp --version` を実行してください。
コマンドが見つからない場合は `npm install -g @marp-team/marp-cli` でインストールしてください。

次に `mmdc --version` を実行してください。
コマンドが見つからない場合は `npm install -g @mermaid-js/mermaid-cli` でインストールしてください。

### Step 0: ファクト収集（文章を書く前に必ず実行）

`templates/thinking-frameworks.md` と `templates/business-context.md` を参照し、ファクト収集と同時にビジネスコンテキストを把握する。

**追加収集項目**:
```yaml
  business_context:
    purpose: "(1-2文でシステムの存在理由)"
    users: [{role: "...", needs: "..."}]
  selected_frameworks: [MECE, "So What? / Why So?"]
```

対象パスのコードを読み、発見した事実を構造化データとして整理してください。
ファクト収集で列挙されていない事実を本文に書くことは禁止です。

````yaml
facts:
  routes:       # ルーティング定義
    - method: (HTTP メソッド)
      path: (パス)
      controller: (コントローラー#メソッド)
      source: "(ファイル:行番号)"
  controllers:  # コントローラー
    - name: (クラス名)
      file: (ファイルパス)
      source: "(ファイル:行番号)"
  models:       # モデル
    - name: (クラス名)
      table: (テーブル名)
      source: "(ファイル:行番号)"
  screens:      # 画面
    - name: (画面名)
      path: (URL パス)
      component: (コンポーネントファイル)
      source: "(ファイル:行番号)"
````

**ルール**: 各エントリの `source`（ファイル:行番号）は必須。推測は `confirmed: false` を付与。
**モジュール有効性チェック（必須）**: `quality-rules.md` §4a に従い各モジュールの active を判定する。コードの存在だけでモジュールを列挙しない。

### Step 0b: ストーリーライン設計

`templates/storyline-design.md` を参照し、スライド全体のストーリーラインを設計する。
メインメッセージを1文で定義し、各スライドのサブメッセージを箇条書きで列挙してから本文を書き始める。

### Step 0c: 重要度マッピング

`templates/priority-analysis.md` を参照し、Step 0 のファクトとビジネスコンテキストに基づいて全機能を Tier 1（コア）/ Tier 2（重要）/ Tier 3（補助）に分類する。
Tier 1 は全機能の 20-30% を目安とする。Tier 1 の機能はビジネスルール詳細・エッジケース・パフォーマンス考慮点まで深掘りし、Tier 3 は概要+API一覧のサマリーに留める。
Tier分類の対象は active: true のモジュールのみ。disabled_modules は「問題点・技術的負債」スライド（第4部 12番）に無効化モジュール一覧として記載する（`quality-rules.md` §4a 参照）。

---

### Step 2: コードを分析してスライド構成を決定

対象パスのコードを**すべて**読み、全機能を網羅的にカバーするスライドを設計してください。
概要だけでなく、このスライドを見ればシステムの全体像と各機能の詳細がわかる「完全版資料」を目指します。

**⚠️ 品質ルール（最優先）:**
`templates/quality-rules.md` の全ルールに従うこと。ソース引用は `(file:line)` 形式（本文に表示）。

**エンジニア向け（20〜30枚）**

第1部：全体像（5枚）
1. タイトル（システム名・一言説明・技術スタック）
2. システム全体像（アーキテクチャ構成図）
3. 技術スタック詳細（バージョン・依存ライブラリ一覧）
4. ディレクトリ構成と責務分担
5. 画面遷移図（stateDiagram で全画面とその遷移を網羅）

第2部：機能カタログ（10〜15枚、全機能を1機能1スライドで）
- 各画面・各機能ごとに1スライドを使う
- 各スライドに含める内容：
  - 機能名・画面名
  - 何ができるか（ユーザー視点の1-2行説明）
  - 関連コンポーネント（Controller → Repository → Model の流れ）
  - 主要APIエンドポイント（メソッド + パス）
  - 重要なビジネスルール・計算ロジック（あれば）

第3部：データフロー（3〜5枚）
6. メインのデータフロー図（リクエスト→レスポンスの全体像）
7. 正常系シーケンス（代表的なユースケース）
8. 異常系シーケンス（認証エラー、バリデーションエラー等）
9. バッチ処理・非同期処理フロー（あれば）
10. データモデル概要（主要テーブルの関係図）

第4部：設計評価（3〜5枚）
11. 設計の良い点
12. 問題点・技術的負債
13. 改善案（優先順位付き）
14. 意思決定ログ（主要な設計判断とその理由）
15. まとめ・次のアクション

**営業向け（10〜15枚）**

**⚠️ 営業向けスライドにも最低4枚のSVG図を含めること。**
図はビジネスフロー・効果比較・機能マップなど、技術用語なしで理解できるものにする。

第1部：概要（3枚）
1. タイトル（ビジネス価値を一言で）
2. 何ができるか（機能マップを図で — SVG必須）
3. 導入メリット（Before/After を図で比較 — SVG必須）

第2部：機能紹介（5〜8枚、主要機能ごとに1スライド）
- 各機能ごとに「何ができるか」「誰が使うか」「どんな画面か（説明文で）」
- **定量効果の記述ルール**: DB実データから算出可能な数値のみ定量表現に使用する。工数削減等は「効果（推定）」と明記し前提条件を併記する。前提不明なら定性表現に留める。改善後をゼロと断言しない
- 技術用語は使わない。ユーザー視点の言葉のみ
- 各機能スライドにも図・テーブル・カードのいずれかを必ず含める

第3部：まとめ（3〜4枚）
4. システム全体の業務フロー図（SVG必須）
5. 定量効果まとめ（テーブルで全機能の Before/After を一覧化）
6. よくある質問と答え
7. まとめ・導入ステップ・次のアクション

**初心者向け（12〜18枚）**

第1部：はじめに（3枚）
1. タイトル（わかりやすい言葉で）
2. そもそも何をするシステムか
3. 登場人物と役割（ユーザー種別・権限の違い）

第2部：機能ツアー（6〜10枚、画面ごとに1スライド）
- 各画面で「何ができるか」をやさしく説明
- 処理の流れを図で示す
- 専門用語には必ず注釈をつける
- 画面遷移図で「どの画面からどこに行けるか」を示す

第3部：まとめ（3〜5枚）
4. 全体の画面遷移図（簡略版）
5. 重要なポイント3〜5つ
6. 用語解説（テーブルにまとめる）
7. まとめ・次に学ぶべきこと

### Step 3: Mermaid 図を SVG に変換

スライドに含める Mermaid 図を個別の `.mmd` ファイルとして書き出し、SVG に変換してください。

```
/tmp/{機能名}-slide-1.mmd  → /tmp/{機能名}-slide-1.svg
/tmp/{機能名}-slide-2.mmd  → /tmp/{機能名}-slide-2.svg
...
```

各 `.mmd` ファイルに対して以下のコマンドを実行してください：

```bash
mmdc -i /tmp/{機能名}-slide-N.mmd -o /tmp/{機能名}-slide-N.svg -t neutral -b transparent
```

**Mermaid 図の品質ルール**
- 実コードの構造を根拠に正確に描くこと
- ノード名はコンポーネント名そのままを使う（省略しない）
- 矢印のラベルで「何が流れるか」を明記する
- 1つの図に詰め込みすぎない（ノード数は最大10〜12）
- 複雑な場合は図を分割する

**⚠️ スライド内画像の切れ防止ルール（最重要）**
- **縦方向のノード数は最大4つ**（flowchart TB / direction TB のステップ数）。5つ以上ある場合はステップを統合して4つ以下にする
- SVG が縦に長くなる図（flowchart TB, sequenceDiagram の参加者5人以上）は、ステップを統合するか `flowchart LR`（横方向）に変更する
- `.fig-text` レイアウト内の SVG + 右カラムの要素を合計5つ以内に抑える（カード3枚 or テーブル4行が上限）
- 生成後に必ず PDF 出力して全ページの下端が切れていないか確認する。切れている場合はノード数を減らして再生成する

### Step 4: Marp用Markdownファイルを生成

テーマCSSは `templates/slides/theme.css` を参照してください。
`/tmp/{機能名}-{対象読者}.md` を以下の形式で生成してください。

```markdown
---
marp: true
theme: default
paginate: true
backgroundColor: "#ffffff"
style: |
  （templates/slides/theme.css の内容をここにコピー）
---

<!-- _class: title -->
# （機能名）
## （一言説明）

---

（以降のスライド）
```

**スライド作成の必須ルール**

1. テキストのみのスライドは絶対に作らない。全スライドに以下のいずれかを含める：
   - SVG画像（Step 3 で生成した Mermaid 図）: `![説明](/tmp/{機能名}-slide-N.svg)`
   - テーブル（最大5行。それ以上は分割）
   - コードブロック（重要ロジックのみ。最大10行）
   - `.columns` クラスの2カラムレイアウト
   - `.card` クラスのポイント強調

2. 図を含むスライドは `.fig-text` レイアウトを使い、左に図・右に説明を配置：
   ```html
   <div class="fig-text">
   <div>

   ![図の説明](/tmp/{機能名}-slide-N.svg)

   </div>
   <div>

   - ポイント1
   - ポイント2
   - ポイント3

   </div>
   </div>
   ```

3. 1スライドの内容量の上限：
   - 箇条書き: 最大5項目
   - テーブル: 最大5行
   - コードブロック: 最大10行
   - 上限を超える場合はスライドを分割する

4. 説明は対象読者に合わせた言葉を使う

5. セクション区切りには `<!-- _class: section-divider -->` を使う

### セルフレビュー（出力前に必ず実行）

`templates/quality-rules.md` §5 のセルフレビューチェックリストを実行すること。

### Step 5: 変換実行

形式に応じて以下のコマンドを実行してください。

まず出力ディレクトリを作成：
```bash
mkdir -p docs/slides
```

HTML（デフォルト）の場合:
```bash
marp /tmp/{機能名}-{対象読者}.md --html -o docs/slides/{機能名}-{対象読者}.html
```

PDFの場合:
```bash
marp /tmp/{機能名}-{対象読者}.md --html --pdf -o docs/slides/{機能名}-{対象読者}.pdf
```

PPTXの場合:
```bash
marp /tmp/{機能名}-{対象読者}.md --html --pptx -o docs/slides/{機能名}-{対象読者}.pptx
```

形式が省略された場合はHTMLで出力してください。

### Step 6: 品質チェック（自動）

生成したスライドの品質を以下の基準で自動チェックしてください。
問題があればMarkdownを修正して Step 5 をやり直してください。

**チェック項目**
- [ ] 全スライドにビジュアル要素（図・表・コード・カード）があるか
- [ ] SVG画像が正しく生成されているか（0バイトのファイルがないか）
- [ ] 1スライドに箇条書き6項目以上ないか
- [ ] 1スライドにテーブル6行以上ないか
- [ ] テキストだけのスライドがないか
- [ ] 対象読者に合わない専門用語がないか（営業・初心者向けの場合）

問題がなければ Step 7 に進んでください。

### Step 6.5: アセット整理（/tmp 画像の移動）

スライド内の画像パスを修正し、docs/ 配下で自己完結するようにする:

1. 生成したスライド Markdown（`/tmp/{機能名}-{対象読者}.md`）内の全画像パスを走査する
2. `/tmp/` を参照している SVG ファイルを `docs/slides/{機能名}/images/` にコピーする:
   ```bash
   mkdir -p docs/slides/{機能名}/images
   cp /tmp/{機能名}-slide-*.svg docs/slides/{機能名}/images/
   ```
3. スライド Markdown 内のパスを相対パス `{機能名}/images/{filename}` に書き換える:
   - `/tmp/{機能名}-slide-N.svg` → `{機能名}/images/{機能名}-slide-N.svg`
4. 書き換え後のスライド Markdown を `/tmp/` から `docs/slides/{機能名}-{対象読者}.md` にもコピーする
5. Marp で再変換する（パス書き換え後に再変換が必須）:
   ```bash
   marp docs/slides/{機能名}-{対象読者}.md --html -o docs/slides/{機能名}-{対象読者}.html
   ```

### Step 7: 中間ファイルのクリーンアップ

スライド生成に使った /tmp の中間ファイルを削除してください：

```bash
rm /tmp/{機能名}-slide-*.mmd /tmp/{機能名}-slide-*.svg
```

Marp ソースファイル（`/tmp/{機能名}-{対象読者}.md`）は他の形式への再変換に使えるため残してください。

### Step 8: 完了報告

```
スライド生成完了
   ファイル：docs/slides/{機能名}-{対象読者}.（拡張子）
   枚数：N枚
   対象読者：{対象読者}
   Mermaid図：N個（SVG変換済み）

他の形式に変換したい場合：
  HTML: marp /tmp/{機能名}-{対象読者}.md --html -o docs/slides/{機能名}-{対象読者}.html
  PDF:  marp /tmp/{機能名}-{対象読者}.md --html --pdf -o docs/slides/{機能名}-{対象読者}.pdf
  PPTX: marp /tmp/{機能名}-{対象読者}.md --html --pptx -o docs/slides/{機能名}-{対象読者}.pptx
  ※ Google スライドへは PPTXをインポートして使えます

生成内容の実在性を検証するには：
  python scripts/verify_docs.py \
    --docs-dir ./docs \
    --source-dir {対象パス} \
    --name {機能名}
```
