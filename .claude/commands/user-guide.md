# /project:user-guide

エンドユーザー向けの操作ガイドを自動生成します。
技術用語を一切使わず、新入社員がこれだけ読めばシステムを操作できるレベルの実用的なマニュアルです。

## 引数

$ARGUMENTS をスペース区切りで以下のように解釈してください：
- 第1引数: 対象パス
- 第2引数: システム名（ファイル名に使用）

```
例: /project:user-guide ./src my-app
例: /project:user-guide /Users/.../my-project my-app
```

以降、第1引数を「対象パス」、第2引数を「システム名」として参照します。

## 実行内容

対象パスのコードを**すべて**読み込み、以下の構成でエンドユーザー操作ガイドを生成してください。

---

### Step 0: ファクト収集（文章を書く前に必ず実行）

`templates/business-context.md` を参照し、ファクト収集と同時にシステムの業務目的とユーザー種別を把握する。

対象パスのコードを読み、全画面の事実を構造化データとして整理してください。
ファクト収集で列挙されていない画面・操作を本文に書くことは禁止です。

````yaml
facts:
  screens:      # 全画面
    - name: (画面名)
      path: (URL パス)
      component: (コンポーネントファイル)
      elements:
        buttons: ["ボタン名", ...]
        forms: [{name: "フォーム名", fields: ["フィールド名", ...]}]
        tables: [{name: "テーブル名", columns: ["カラム名", ...]}]
      source: "(ファイル:行番号)"
  routes:       # 関連ルート
    - path: (パス)
      source: "(ファイル:行番号)"
````

**ルール**: テンプレート（Vue `<template>`, JSX, Blade等）を実際に読み、ボタン名・フィールド名・カラム名を正確に抽出する。
**モジュール有効性チェック（必須）**: `quality-rules.md` §4a に従い各モジュールの active を判定する。コードの存在だけでモジュールを列挙しない。active: true の機能のみ記載し、disabled_modules は出力しない（エンドユーザー向けのため）。

---

### Step 0b: ストーリーライン設計

`templates/storyline-design.md` を参照し、「この操作ガイドで読者に最も伝えたいこと」を1文で定義する。
日常業務で最も使う機能を先頭に配置し、使用頻度順に章を構成する。

---

### ⚠️ 共通ルール（最優先）

`templates/quality-rules.md` の全ルールに従うこと。ソース引用は `<!-- source: file:line -->` 形式（HTMLコメント、不可視）。
以下は本コマンド固有の追加ルール：

1. **技術用語は一切使わない**: Controller, Repository, API, JWT, Vuex, Eloquent, ORM, Middleware 等は禁止。ユーザーに見える言葉だけを使う
2. **です/ます調**で統一する
3. 全ての操作を**番号付きステップ**（1. → 2. → 3.）で記述する
4. 画面要素は「」で囲む: 「ログイン」ボタン、「月選択」ドロップダウン、「売上一覧」テーブル
5. 適切な箇所に **💡 ポイント** と **⚠️ 注意** を挿入する
6. **全画面・全操作を網羅する**（主要画面だけでなく、全画面を対象にする）
7. **操作フローの検証**: 「ボタンをクリックすると〇〇が起きる」と書く場合、イベントハンドラ → API呼び出し → 結果表示の流れをコードで確認する
8. 先頭にヘッダー情報を含める:
   ```
   > 生成日: {今日の日付} | 対象: {システム名} 操作ガイド | ツール: knowledge-asset-tool
   ```
9. **ページ内アンカーリンク目次は含めない**（MkDocs Material の右サイドバーが自動生成するため二重になる）

---

### 構成テンプレート

以下の章構成に従って生成してください。章の数・内容はシステムの画面数に応じて調整してください。

#### 第1章: はじめに
- このシステムの目的（2-3行、平易な言葉で）
- 利用環境（推奨ブラウザ、画面サイズ等）
- 画面の基本構成（ヘッダー、サイドバー、メインエリアの役割）
- 共通操作の説明（テーブルのソート、ページ切替、検索、ダイアログの開閉）

#### 第2章: ログイン・パスワード
- ログイン手順
- パスワードを忘れた場合
- パスワード変更
- ログアウト

#### 第3章〜: 各画面の操作（1画面1章 or 関連画面をグループ化）

**各画面の章に含める内容（全項目必須）:**

**画面の目的**: この画面で何ができるか（ユーザー視点で2-3行）

**画面の開き方**: サイドバーのどこをクリックするか

**画面の見方**: 画面に表示される要素の説明
- フィルター・検索エリアの使い方
- テーブル・カード・グラフの見方
- ボタンの位置と役割

**基本操作**: ステップバイステップで記述

```
### データを新しく登録する

1. 画面右上の「新規登録」ボタンをクリックします
2. 入力画面（ダイアログ）が表示されます
3. 以下の項目を入力します：
   - 「案件」: ドロップダウンから選択します
   - 「日付」: カレンダーから選択します
   - 「金額」: 半角数字で入力します
4. 入力内容を確認し、「保存」ボタンをクリックします
5. 一覧に新しいデータが追加されます

💡 ポイント: 金額は小数点以下2桁まで入力できます
⚠️ 注意: 保存後も「確定」するまでは仮の状態です
```

**この画面でできる全ての操作一覧**: テーブルで整理

| 操作 | 手順の概要 | 必要な権限 |
|------|----------|----------|

#### 権限（ロール）早見表の章

システムのロール（役割）ごとに「できること/できないこと」をマトリクス表で整理。
技術用語は使わず「○ できる」「× できない」「△ 閲覧のみ」で表現。

| 操作 | 管理者 | 拠点長 | 営業 | 経理 | スタッフ | ... |
|------|--------|--------|------|------|---------|-----|

#### FAQ の章

実際の業務で発生しうる質問を10-15個。Q&A形式。

例:
- ログインできない場合は？
- 入力を間違えた場合は？
- 確定した売上を修正したい場合は？
- 画面が表示されない場合は？
- データは削除されるのか？

#### 業務フロー早見表の章

主要な業務フローをテキストベースの図で表現：
- 日次フロー（毎日やること）
- 月次フロー（月末にやること）
- 新規登録フロー（初期設定でやること）

```
日次フロー:
ログイン → ダッシュボードで状況確認 → 売上入力 → 確認

月次フロー:
売上データ確認 → 売上確定 → 請求書作成 → PDF出力 → 送付
```

### セルフレビュー（出力前に必ず実行）

`templates/quality-rules.md` §5 のセルフレビューチェックリストを実行すること（論理的深み・ビジネス接続チェックは不要）。

---

## 出力形式

以下の形式で出力してください。

```
--- FILE: docs/manual/{システム名}/09-user-guide.md ---
（エンドユーザー操作ガイド全文）
```

出力後、以下を案内してください：
```
操作ガイド生成完了！

  対象: {システム名}
  章数: N章
  画面数: N画面の操作手順を網羅

  次のコマンドでファイルに保存できます：
  python scripts/save_output.py \
    --name {システム名} \
    --output-dir ./docs

  生成内容の実在性を検証するには：
  python scripts/verify_docs.py \
    --docs-dir ./docs \
    --source-dir {対象パス} \
    --name {システム名}
```
