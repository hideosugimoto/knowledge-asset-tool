# 分析思考フレームワーク

本ファイルは、コード分析の前にシステム特性を見極め、最適な思考フレームワークを選択するためのガイドである。

## 1. フレームワークカタログ

### MECE（漏れなくダブりなく）
- **目的**: 機能・コンポーネント・データの分類に漏れとダブりがないか検証する
- **適用条件**: 全システム、全コマンドで常時適用
- **出力への影響**: 一覧表の作成時に「この分類で全てカバーできているか？ 重複はないか？」を検証。機能カタログ、API一覧、テーブル一覧に特に適用
- **チェック方法**: 列挙した項目の合計がファクト収集の総件数と一致するか確認

### So What? / Why So?
- **目的**: 各記述に「だから何？（So What?）」と「なぜそう言える？（Why So?）」を問い、論理的深みを担保する
- **適用条件**: 全システム、全コマンドで常時適用
- **出力への影響**: 事実の列挙で終わらせず、必ず「意味」を付与する
  - 悪い例: 「このシステムは Laravel + Vue.js で構成されている」
  - 良い例: 「このシステムは Laravel + Vue.js で構成されている。サーバーサイドレンダリングではなく SPA 構成を採用しており、これによりフロントエンドの独立したデプロイが可能だが、初期表示速度とSEOにはトレードオフがある」
- **チェック方法**: 各セクションの主要な記述に対して「だから何？」と問い、答えが書かれているか確認

### イシューツリー
- **目的**: 問題やリスクを構造的に分解し、根本原因と影響範囲を明確にする
- **適用条件**: リスク分析・技術的負債の評価時に適用
- **出力への影響**: 「リスクと技術的負債」セクションでは、単なるリスト列挙ではなく、問題の構造を示す
  - 悪い例: 「N+1クエリがある」「テストがない」「バージョンが古い」
  - 良い例: 問題を「パフォーマンス」「保守性」「セキュリティ」のツリーに分解し、各問題の因果関係と影響範囲を示す
- **適用コマンド**: analyze, analyze-full, manual の §8/§11/第8章

### 4+1 Architectural View Model
- **目的**: アーキテクチャを5つの視点から多角的に記述する
- **適用条件**: 中〜大規模システム（画面15+、テーブル30+）
- **5つのビュー**:
  - 論理ビュー: コンポーネントの責務と関係（→ §5 ビルディングブロック）
  - プロセスビュー: 並行性、同期、非同期処理（→ §6 ランタイム）
  - 物理ビュー: デプロイ構成（→ §7 デプロイ）
  - 開発ビュー: コードの構成、パッケージ構造（→ §2 制約、ディレクトリ構成）
  - ユースケースビュー: 主要なユースケース（→ §6 ランタイム、第7章 ウォークスルー）
- **適用コマンド**: analyze, analyze-full, manual

### ATAM（Architecture Tradeoff Analysis Method）
- **目的**: アーキテクチャの品質属性間のトレードオフを明示する
- **適用条件**: 複雑なシステム（ビジネスロジック複雑、非同期処理あり、外部連携多数）
- **出力への影響**: 品質要件セクションに追加する項目
  - 感度ポイント: 1つの設計判断が複数の品質属性に影響する箇所
  - トレードオフポイント: ある品質属性の改善が別の品質属性を犠牲にする箇所
  - リスク: 感度ポイントかつ未解決の設計課題
- **適用コマンド**: analyze-full（§10 品質要件）, manual（第8章 設計評価）

### SWOT
- **目的**: システムの内部要因（強み/弱み）と外部要因（機会/脅威）を整理する
- **適用条件**: 設計評価・改善案セクションで適用
- **出力への影響**:
  - 強み（S）: 良い設計判断、適切な技術選定
  - 弱み（W）: 技術的負債、設計上の問題
  - 機会（O）: 改善によって得られるメリット、新技術への移行余地
  - 脅威（T）: EOLリスク、セキュリティ脆弱性、スケーラビリティ限界
- **適用コマンド**: analyze, analyze-full, manual

### ビジネスモデルキャンバス
- **目的**: 技術をビジネス価値に翻訳し、非技術者にもわかる形で整理する
- **適用条件**: 営業向け・ビジネス向け資料で適用
- **9つの構成要素**:
  - 顧客セグメント: 誰が使うか
  - 価値提案: 何を解決するか
  - チャネル: どう届けるか
  - 顧客関係: どう維持するか
  - 収益の流れ: どう収益化するか
  - 主要リソース: 何が必要か
  - 主要活動: 何をするか
  - パートナー: 誰と組むか
  - コスト構造: 何にコストがかかるか
- **適用コマンド**: analyze-share, analyze-slide（営業向け）, analyze-full（C-2 営業向け）

## 2. 選択マトリクス

### システム特性の判定基準

| 特性 | small | medium | large |
|------|-------|--------|-------|
| 画面数 | < 10 | 10-30 | 30+ |
| テーブル数 | < 20 | 20-50 | 50+ |
| APIエンドポイント数 | < 30 | 30-80 | 80+ |
| ソースファイル数 | < 50 | 50-200 | 200+ |

| 特性 | simple | moderate | complex |
|------|--------|----------|---------|
| ビジネスロジック | CRUD中心 | 計算・条件分岐あり | 複雑なワークフロー |
| 非同期処理 | なし | キュー/バッチあり | イベント駆動/CQRS |
| 外部連携 | DBのみ | 1-2外部API | 3+外部サービス |
| 認証・認可 | 単純 or なし | ロールベース | 細粒度権限制御 |

### フレームワーク選択ガイド

**常時適用（全システム）:**
- MECE
- So What? / Why So?

**規模・複雑度に応じて追加:**

| システム特性 | 追加フレームワーク |
|------------|----------------|
| medium以上 | 4+1 Architectural View |
| complex | ATAM |
| 業務系 medium以上 | SWOT |
| 営業向け資料を含む | ビジネスモデルキャンバス |
| リスク分析が必要 | イシューツリー |

## 3. コマンド別デフォルト推奨

| コマンド | 常時適用 | 条件付き追加 |
|---------|---------|------------|
| analyze | MECE, So What? | +4+1(medium以上), +SWOT(medium以上), +イシューツリー |
| analyze-full | MECE, So What? | +4+1(medium以上), +ATAM(complex), +SWOT, +イシューツリー |
| manual | MECE, So What? | +4+1(medium以上), +ATAM(complex) |
| analyze-slide | MECE, So What? | +ビジネスモデルキャンバス(営業向け), +4+1(エンジニア向けmedium以上) |
| analyze-share | MECE, So What? | +ビジネスモデルキャンバス |
| analyze-rag | MECE, So What? | （追加なし — 均等チャンクが前提） |
| user-guide | MECE | （So What?は不要 — 操作手順は事実の記述） |
| quick-ref | MECE | （So What?は不要 — 早見表は簡潔さが優先） |

## 4. 悪い例と良い例

### テンプレート埋め（悪い例）
```
§1. はじめに
このシステムはLaravel + Vue.jsで構成されたWebアプリケーションです。
主要な機能は以下の通りです：
- ユーザー管理
- 売上管理
- 請求管理
```

### フレームワーク適用済み（良い例）
```
§1. はじめに
このシステムは、月次の売上集計と請求処理を自動化し、
経理部門の手作業を年間約500時間削減することを目的とした業務システムである。
[So What?: 手作業削減がビジネス価値の核心]

主要な機能要件は以下の3層に分類される（MECE）：
- データ入力層: 売上データの登録・修正（日次業務の入口）
- 処理・集計層: 月次集計・請求書生成（ビジネスロジックの中核）
- 出力・連携層: PDF出力・外部システム連携（業務の出口）
[Why So?: ルーティング定義とコントローラーの責務分析から3層に分類]
```
