# ビジネスコンテキスト収集ガイド

コード分析の前に「誰が何のために使っているか」を理解するためのガイド。
技術的に正確でもビジネス的に的外れな記述を防ぐ。

## 1. 自動収集チェックリスト

以下のソースから順番に情報を収集する。各項目で見つかった情報は `business_context` YAML に記録する。

### 優先度: 高（必ずチェック）
- [ ] **README.md / README**: 冒頭の説明文、「About」セクション
- [ ] **package.json / composer.json**: `description`, `name`, `keywords` フィールド
- [ ] **ロール・権限定義**: Enum, 定数, ミドルウェアからユーザー種別を特定
  - 例: `admin`, `manager`, `staff`, `viewer` → 管理者・拠点長・スタッフ・閲覧者
- [ ] **ルーティングプレフィックス**: `/admin`, `/api/v1`, `/portal`, `/client` → ユーザー種別の手がかり

### 優先度: 高（モジュール有効性）
- [ ] **有効/無効モジュール定義**: DB設定テーブル、設定ファイル、ルーティング登録で各モジュールの active 状態を確認（`quality-rules.md` §4a 参照）
  - 例: `vtiger_tab WHERE presence=1`, `config/app.php` の providers, `INSTALLED_APPS`, `routes/` のルーティング登録

### 優先度: 中（あれば確認）
- [ ] **設定ファイル**: `.env.example`, `config/` 配下 → 環境名、サービス名、外部連携先
- [ ] **i18n ファイル**: `lang/`, `locales/` → 対象言語・市場
- [ ] **Docker / docker-compose**: サービス構成、インフラの概要
- [ ] **ソースコードのコメント**: ファイルヘッダー、クラスの docblock → ビジネス目的の記述

### 優先度: 低（補助的）
- [ ] **テーブル名・カラム名**: ドメイン用語の手がかり（`invoices`, `deliveries`, `patients` 等）
- [ ] **CI/CD設定**: デプロイ先、環境数 → 利用規模の推測

## 2. 質問テンプレート

自動収集で confidence が low の場合、以下から最大3問を選んでユーザーに質問する。
自動収集で推測できた項目はスキップする。

| # | 質問 | 目的 | スキップ条件 |
|---|------|------|-----------|
| Q1 | このシステムの主なユーザーは誰ですか？（例: 社内の経理部門、外部の取引先、エンドユーザー） | ステークホルダー特定 | ロール定義から推測可能 |
| Q2 | このシステムが解決している最大の課題は何ですか？ | ビジネス価値特定 | README に明記されている |
| Q3 | 利用規模は？（ユーザー数・データ量の概算） | スケール推測 | Docker/インフラ構成から推測可能 |
| Q4 | 最も頻繁に使われる機能はどれですか？ | 重要度判定の入力 | 画面構成から推測可能 |
| Q5 | 競合システムや置き換え対象はありますか？ | 外部コンテキスト | 通常不明（低優先） |

**質問のルール**:
- 1度に3問以上は聞かない（ユーザー体験を損なう）
- 自動収集で confidence: medium 以上なら質問不要
- 答えが得られなくても分析は進める（推測ベースで続行し【推測】と明記）

## 3. 出力フォーマット

```yaml
business_context:
  purpose: "(1-2文でシステムの存在理由)"
  domain: "(業務ドメイン: 会計、物流、人事、EC、医療、教育等)"
  users:
    - role: "(ユーザー種別)"
      needs: "(そのユーザーが求めるもの)"
      source: "(推測根拠: README / ロール定義 / ユーザー回答)"
  business_value: "(このシステムのビジネス上の価値)"
  scale:
    estimated_users: "(推測ユーザー数)"
    data_volume: "(推測データ量)"
  external_context:
    competitors: "(競合/置換対象)"
    integrations: "(外部連携先)"
  confidence: low|medium|high
  sources_used: ["README.md", "composer.json", "app/Models/User.php", ...]
```

## 4. ビジネスコンテキストの活用先

収集したビジネスコンテキストは以降の全ステップで参照する：

| 活用先 | 使い方 |
|--------|-------|
| §1 はじめに | システム目的・ステークホルダーの記述根拠 |
| ストーリーライン設計 | メインメッセージの構築根拠 |
| 重要度マッピング | コア機能の判定根拠 |
| 営業/ビジネス向け資料 | ビジネス価値の記述根拠 |
| 設計評価 | 「ビジネス要件に対して適切か」の判断根拠 |
| user-guide | 業務フロー早見表の構築根拠 |
