# /project:quick-ref

A4 1枚に収まるクイックリファレンスカードを自動生成します。
印刷してデスクに貼れる早見表です。操作ガイド（user-guide）の情報を凝縮した実用資料。

## 引数

$ARGUMENTS をスペース区切りで以下のように解釈してください：
- 第1引数: 対象パス
- 第2引数: システム名（ファイル名に使用）

```
例: /project:quick-ref ./src my-app
例: /project:quick-ref /Users/.../my-project my-app
```

以降、第1引数を「対象パス」、第2引数を「システム名」として参照します。

## 実行内容

対象パスのコードを読み込み、以下の4セクションからなるクイックリファレンスカードを生成してください。
**A4 1枚に印刷できる分量**に収めることが最優先の制約です。

---

### Step 0: ファクト収集（文章を書く前に必ず実行）

`templates/business-context.md` を参照し、ファクト収集と同時にシステムの業務目的とユーザー種別を把握する。

対象パスのコードを読み、全画面の事実を構造化データとして整理してください。
ファクト収集で列挙されていない画面・操作を本文に書くことは禁止です。

````yaml
facts:
  screens:      # 全画面
    - name: (画面名)
      path: (URL パス)
      component: (コンポーネントファイル)
      elements:
        buttons: ["ボタン名", ...]
        forms: [{name: "フォーム名", fields: ["フィールド名", ...]}]
        tables: [{name: "テーブル名", columns: ["カラム名", ...]}]
      source: "(ファイル:行番号)"
  routes:       # 関連ルート
    - path: (パス)
      source: "(ファイル:行番号)"
````

**ルール**: テンプレート（Vue `<template>`, JSX, Blade等）を実際に読み、ボタン名・フィールド名・カラム名を正確に抽出する。
**モジュール有効性チェック（必須）**: `quality-rules.md` §4a に従い各モジュールの active を判定する。コードの存在だけでモジュールを列挙しない。active: true の機能のみ記載し、disabled_modules は出力しない（A4 1枚の制約上）。

---

### ⚠️ 共通ルール（最優先）

`templates/quality-rules.md` の全ルールに従うこと。ソース引用は `<!-- source: file:line -->` 形式（HTMLコメント、不可視）。
以下は本コマンド固有の追加ルール：

1. 技術用語は使わない（エンドユーザーが読む前提）
2. 全体でA4 1枚（Markdown で約80〜120行）に収める。溢れる場合は使用頻度の高い操作を優先する
3. 先頭にヘッダー情報を含める:
   ```
   > 生成日: {今日の日付} | 対象: {システム名} クイックリファレンス | ツール: knowledge-asset-tool
   ```

---

### セクション1: よく使う操作 TOP10

コードから以下を分析し、使用頻度の高い操作を10件選定する：
- ルーティング定義（routes, pages, router 等）から全画面を把握
- 各画面のメインアクション（CRUD操作、ボタン、フォーム送信）を特定
- ナビゲーション/サイドバーの並び順（上にあるものほど頻繁に使う傾向）

テーブル形式で出力：

| # | やりたいこと | 画面 | 操作 |
|---|------------|------|------|
| 1 | （ユーザー視点の目的） | （画面名） | 「（ボタン名/メニュー名）」 |

**選定基準（優先順）：**
1. ナビゲーションのトップレベルに表示される機能
2. CRUD の Create（登録系）操作
3. 一覧検索・フィルター操作
4. 出力・エクスポート操作（PDF出力、CSV出力等）

### セクション2: 画面マップ

サイドバー/ナビゲーションの構造をそのまま反映した画面一覧。
コードのルーティング定義・メニュー定義から正確に抽出する。

```
（アイコンがあればそのまま使う。なければ省略）
ダッシュボード ── 状況確認・通知
スタッフ管理 ── 登録・編集・権限設定
  └ 詳細画面 ── 個別情報・履歴
配送案件 ── 案件作成・ステータス更新
  └ 案件詳細 ── 工程管理・配車
...
```

- ネスト構造はルーティングの親子関係を反映する
- 各画面の右に主な用途を1行で添える
- 画面数が15を超える場合は主要画面のみに絞り「他N画面」と注記

### セクション3: 権限早見表

コードのミドルウェア・ガード・ロール定義から正確に抽出する。

| 操作 | （ロール1） | （ロール2） | （ロール3） | ... |
|------|-----------|-----------|-----------|-----|
| （操作名） | ○ | △ | × | ... |

- ○ = 全操作可能 / △ = 閲覧のみ / × = アクセス不可
- ロール名はコードで定義されている名称を使う（日本語があれば日本語優先）
- 操作は画面単位でグループ化する
- ロール定義が存在しない場合は「権限管理なし（全ユーザー同一権限）」と明記

### セクション4: 困ったときは

コードのエラーハンドリング・バリデーション・認証処理から抽出する。

| 症状 | 原因 | 対処 |
|------|------|------|
| ログインできない | パスワード誤り / アカウントロック | （コードから判断できる対処法） |
| 保存ボタンが押せない | 必須項目が未入力 | （バリデーションルールから特定） |
| データが表示されない | 期間フィルター / 権限不足 | （フィルター初期値やガード条件から特定） |

- コードで確認できるエラーケースのみ記載（5〜8件程度）
- 問い合わせ先がコード内に定義されていれば末尾に記載
- 問い合わせ先が不明な場合は「システム管理者にお問い合わせください」とする

## 出力形式

以下の形式で出力してください。「{システム名}」は第2引数の値に置き換えてください。

```
--- FILE: docs/manual/{システム名}/10-quick-reference.md ---
（クイックリファレンスカード全文）
```

出力後、以下を案内してください：
```
クイックリファレンス生成完了！

  対象: {システム名}
  セクション: 4（よく使う操作 / 画面マップ / 権限早見表 / 困ったときは）
  想定印刷: A4 1枚

  次のコマンドでファイルに保存できます：
  python scripts/save_output.py \
    --name {システム名} \
    --output-dir ./docs

  生成内容の実在性を検証するには：
  python scripts/verify_docs.py \
    --docs-dir ./docs \
    --source-dir {対象パス} \
    --name {システム名}

  💡 PDF で印刷用に出力するには：
  1. MkDocs サイトで該当ページを開く
  2. ブラウザの印刷機能で PDF 保存（用紙: A4、余白: 最小）
```
